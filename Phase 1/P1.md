# P1 — Authentication & Login Security Learn Notes

## Amaç

Bu bölümde Phase 1 için önce kavram tarafını toparladım. Daha uygulama yazma veya v1/v2 karşılaştırmasına girmedim. Amacım login güvenliğinde hangi kararın client tarafında, hangi kararın server tarafında verilmesi gerektiğini anlamak.

Bu kısımda özellikle şunlara baktım:

- Login kararını neden server vermeli?
- Access token ve refresh token farkı ne?
- JWT nedir, opaque token'dan farkı ne?
- Encode, encrypt, hash ve sign farkı ne?
- Token'lar Android içinde nerede saklanmalı?
- Android Keystore bu işin neresinde?
- Biyometriyi sadece ekran açmak için kullanmak neden zayıf?
- Brute-force, rate limiting ve user enumeration ne demek?
- Logout sadece client'tan token silmek mi, yoksa server tarafında da bir şey yapılmalı mı?

## Server-authoritative Login

Login konusunda ilk öğrendiğim şey şu oldu: client hiçbir zaman kendi kendine “ben login oldum” kararını vermemeli.

Mobil uygulama sadece kullanıcıdan username/password alır ve bunu server'a gönderir. Asıl karar server tarafında verilir. Çünkü kullanıcı gerçekten var mı, şifre doğru mu, hesap kilitli mi, token üretilecek mi gibi kararların hepsi backend'in kontrolünde olmalı.

Yanlış düşünce şuna benziyor:

```text
Kullanıcı doğru şifreyi girdi gibi görünüyor.
O zaman client içinde isLoggedIn = true yapıp ekranı açayım.
```

Bu zayıf bir yaklaşım. Çünkü Android app saldırganın cihazında çalışıyor. Saldırgan uygulamayı decompile edebilir, runtime'da Frida ile fonksiyonları hook'layabilir veya client içindeki boolean değerleri manipüle edebilir.

Daha doğru mantık şu:

```text
Client login isteği atar.
Server kullanıcıyı doğrular.
Server geçerli token üretir.
Client bu token ile sonraki API isteklerini yapar.
Server her istekte token'ı doğrular.
```

Yani client sadece server'ın verdiği sonuca göre hareket eder. Son güvenlik kararı server'dadır.

Gerçek hayatta bunu banka uygulaması gibi düşünebilirim. Uygulamanın ekranda “giriş başarılı” demesi tek başına hiçbir şey ifade etmez. Banka backend'i gerçekten geçerli oturum vermediyse bakiye, transfer, profil gibi hiçbir endpoint çalışmamalı.

## Token-based Auth

Token-based auth'ta kullanıcı her istekte tekrar username/password göndermez. Kullanıcı bir kere login olur, server ona token verir, client da sonraki isteklerde bu token'ı kullanır.

Genel akış şöyle:

```text
1. Client username/password ile login isteği atar.
2. Server bilgileri doğrular.
3. Server access token ve refresh token döner.
4. Client access token ile korumalı endpoint'lere istek atar.
5. Access token süresi bitince refresh token ile yeni access token alınır.
```

Örnek login response:

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIs...",
  "refreshToken": "sample-refresh-token",
  "expiresIn": 900
}
```

Sonraki isteklerde access token genelde `Authorization` header içinde taşınır:

```http
GET /balance HTTP/1.1
Host: api.mockbank.local
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

## Access Token vs Refresh Token

Access token kısa ömürlü token'dır. Genelde API isteklerinde kullanılır.

```text
Access token → kısa ömürlü → API çağrılarında kullanılır
```

Refresh token daha uzun ömürlüdür. Access token süresi bitince yeni access token almak için kullanılır.

```text
Refresh token → uzun ömürlü → yeni access token almak için kullanılır
```

Burada kritik fark şu: access token çalınırsa saldırgan kısa süreli erişim elde eder. Mesela access token 15 dakika geçerliyse risk 15 dakika ile sınırlı kalabilir.

Ama refresh token çalınırsa durum daha tehlikeli olur. Çünkü saldırgan refresh token geçerli kaldığı sürece tekrar tekrar yeni access token alabilir. Yani saldırgan kullanıcının şifresini bilmeden oturumu devam ettirebilir.

Bu yüzden refresh token'ı daha hassas veri gibi düşünmek gerekiyor.

## Secure Refresh Flow

Access token kısa ömürlü olduğu için bir noktada süresi biter. Uygulama bu durumda kullanıcıyı sürekli tekrar login ekranına atmak yerine refresh token ile yeni access token alabilir.

Akış şöyle:

```text
Access token expire olur.
Client /refresh endpoint'ine refresh token gönderir.
Server refresh token'ı doğrular.
Geçerliyse yeni access token döner.
Geçersizse kullanıcı tekrar login olmalıdır.
```

Burada dikkat edilmesi gereken nokta şu: refresh token da server tarafında kontrol edilmelidir. Logout olmuş, süresi dolmuş veya revoke edilmiş refresh token ile yeni access token üretilmemelidir.

## OAuth Nedir?

OAuth bir şirket değil, bir standart/protokoldür. HTTP nasıl bir protokolse OAuth da yetkilendirme için kullanılan bir protokol gibi düşünülebilir.

Google ile giriş yap, GitHub ile giriş yap, Facebook ile giriş yap gibi akışlarda genelde OAuth veya OAuth tabanlı sistemler kullanılır.

Benim burada çıkardığım sonuç şu: OAuth tek başına “güvenlik ürünü” değil; doğru uygulanması gereken bir standart. Yanlış token saklama, zayıf redirect kontrolü veya kötü server doğrulaması varsa OAuth kullanıyorum demek tek başına güvenli yapmaz.

## JWT ve Opaque Token Farkı

Token deyince iki farklı yaklaşım var.

### Opaque Token

Opaque token anlamsız bir string gibi görünür:

```text
a8f3k29x9sLm01...
```

Bu token'ın içinde okunabilir bilgi yoktur. Server bu string'i kendi tarafında bir kayıtla eşleştirir. Saldırgan token'ı görürse içinden user id, role veya expire bilgisi okuyamaz. Sadece token'ı kullanmayı deneyebilir.

### JWT

JWT ise self-contained yapıdadır. İçinde header, payload ve signature vardır:

```text
header.payload.signature
```

Payload kısmında user id, role, expire time gibi bilgiler olabilir:

```json
{
  "sub": "123",
  "role": "user",
  "exp": 1719999999
}
```

JWT'nin önemli noktası şu: JWT çoğu zaman şifreli değildir, sadece encode edilmiş ve imzalanmıştır. Yani saldırgan token'ı ele geçirirse payload kısmını okuyabilir. Ama imza için kullanılan secret/private key'i bilmeden token'ı güvenli şekilde değiştiremez.

Bu yüzden JWT içine gizli bilgi koymamak gerekir. Kullanıcının şifresi, kredi kartı, private bilgi gibi şeyler JWT payload içinde olmamalı.

## Encode, Encrypt, Hash, Sign

Bu kavramları karıştırmamak lazım.

### Encode

Encode şifreleme değildir. Sadece format dönüştürmedir. Base64 buna örnektir.

```text
Encode edilen veri herkes tarafından decode edilebilir.
```

JWT'nin header ve payload kısmı genelde Base64URL ile encode edilir. Bu yüzden jwt.io gibi bir yere yapıştırınca okunabilir JSON olarak görülür.

### Encrypt

Encrypt gerçek gizlilik sağlar. Bir anahtar ile veri şifrelenir, anahtar olmadan veri okunamaz.

```text
Encrypt → gizlilik sağlar
```

Normal JWT yani JWS şifreli değildir. JWT'nin şifreli hali JWE olarak geçer ama çoğu sistemde kullanılan token JWS'tir.

### Hash

Hash tek yönlü özet üretir. MD5 veya SHA gibi fonksiyonlar buna örnektir.

```text
Hash → geri çevrilmez
```

Ama MD5 günümüzde güvenli kabul edilmez. Ayrıca tek başına hash ile JWT imzasını düşünmemek gerekiyor.

### Sign

Sign yani imzalama, verinin değiştirilmediğini kanıtlamak için kullanılır.

JWT'de signature kısmı bunun içindir. HMAC-SHA256 kullanılıyorsa secret key gerekir. RSA/ECDSA kullanılıyorsa private key ile imzalanır, public key ile doğrulanır.

Benim için en net fark şu oldu:

```text
Encode: herkes çözer.
Encrypt: sadece anahtarı olan çözer.
Hash: geri çevrilmez özet üretir.
Sign: veri değişmiş mi değişmemiş mi doğrular.
```

## Token'lar Android'de Nerede Saklanmalı?

Token'ı düz `SharedPreferences` içine yazmak güvenli değildir.

`SharedPreferences` Android'de basit key-value saklama yöntemidir. Dosya olarak uygulamanın private alanında durur:

```text
/data/data/com.example.app/shared_prefs/auth.xml
```

Düz kullanılırsa dosya içeriği şöyle olabilir:

```xml
<map>
    <string name="access_token">eyJhbGciOiJIUzI1NiIs...</string>
    <string name="refresh_token">sample-refresh-token</string>
</map>
```

Normal cihazda Android sandbox başka uygulamaların bu dosyaya erişmesini engeller. Ama rootlu cihazda saldırgan sandbox sınırını aşabilir ve bu XML dosyasını okuyabilir.

Bu yüzden token'lar Keystore-backed encryption ile saklanmalıdır. Pratikte bunun için `EncryptedSharedPreferences`, encrypted DataStore veya doğrudan Android Keystore kullanılabilir.

## EncryptedSharedPreferences

`EncryptedSharedPreferences`, normal `SharedPreferences` gibi çalışır ama veriyi dosyaya yazmadan önce şifreler.

Yani düz XML yerine şöyle anlamsız değerler görünür:

```xml
<map>
    <string name="Aq83kLm...">AV9xQ2xkT9aPq21...</string>
</map>
```

Burada hem key adı hem value okunamaz hale gelir. Saldırgan dosyayı ele geçirse bile plaintext token göremez.

Ben bunu şöyle düşünüyorum: SharedPreferences normal defter gibi. EncryptedSharedPreferences ise kilitli defter gibi. Ama kilidin anahtarını da aynı defterin arasına koymuyor; anahtar Android Keystore içinde tutuluyor.

## Android Keystore — Şifreleme Bağlamında

Phase 0'da Keystore'u uygulama imzalama bağlamında öğrenmiştim. Burada aynı isim farklı bir amaçla kullanılıyor: şifreleme anahtarı koruma.

Normal şifrelemede sorun şudur:

```text
Token'ı şifreledim.
Peki şifreleme anahtarını nerede saklayacağım?
```

Eğer anahtarı koda gömersem veya SharedPreferences'a yazarsam saldırgan hem şifreli veriyi hem de anahtarı bulabilir. Bu durumda şifrelemenin anlamı azalır.

Android Keystore burada anahtar kasası gibi çalışır. Anahtar Keystore içinde üretilir ve normal uygulama dosya alanına çıkmaz. Uygulama anahtarı doğrudan alamaz; sadece “bu veriyle şifreleme/decrypt işlemi yap” diye Keystore'a istek gönderir.

Basit akış:

```text
Keystore içinde AES key oluşturulur.
Token bu key ile şifrelenir.
Ciphertext dosyaya yazılır.
Key dosyada değil, Keystore içinde kalır.
```

Telefon çalınırsa veya saldırgan dosyayı okursa sadece ciphertext görür. Anahtar dosya sisteminde olmadığı için direkt alınamaz.

## TEE ve StrongBox

Keystore bazı cihazlarda donanımsal olarak korunabilir.

TEE, ana işlemcinin izole güvenli alanıdır. StrongBox ise bazı cihazlarda bulunan ayrı güvenlik çipi gibi düşünülebilir.

```text
TEE → izole güvenli çalışma alanı
StrongBox → ayrı donanımsal güvenlik modülü
```

StrongBox varsa anahtarın cihazdan çıkarılması daha da zorlaşır. Rootlu cihaz bile normal şartlarda StrongBox içindeki private key'i doğrudan alamaz.

Ama burada dürüst olmak gerekiyor: rootlu cihaz tehdidi yine çok güçlüdür. Saldırgan Keystore anahtarını çalmak yerine, uygulamanın token'ı decrypt ettiği anı Frida ile yakalamaya çalışabilir. Yani Keystore'u kırmak yerine uygulamanın aldığı sonucu çalar.

Bu yüzden Keystore disk güvenliği için çok değerli ama runtime saldırılarına karşı tek başına yeterli değildir.

## Token Storage'ın Sınırı

EncryptedSharedPreferences veya Keystore-backed storage disk üzerindeki veriyi korur. Ama uygulama token'ı API isteğinde kullanacağı anda token RAM'de plaintext hale gelir.

Çünkü client `Authorization: Bearer ...` header'ını oluşturmak için token'ı bir noktada açık halde kullanmak zorunda.

Rootlu cihazda Frida ile saldırgan bu anı yakalayabilir:

```text
Diskte token şifreli.
Uygulama token'ı decrypt etti.
Token RAM'de plaintext oldu.
Frida ile o değer okunabilir.
```

Bu EncryptedSharedPreferences tamamen boşa gidiyor demek değildir. Çünkü diskten dosya çekme, cihaz hırsızlığı ve basit root dosya okuma saldırılarına karşı hâlâ koruma sağlar. Sadece runtime saldırısına karşı başka katmanların da gerektiğini gösterir.

Benim buradaki kısa sonucum şu:

```text
Encrypted storage disk güvenliğidir.
Runtime güvenliği değildir.
```

## Biometric Auth

Biyometriyi doğru anlamak gerekiyor. Parmak izi veya yüz tanıma tek başına login sistemi değildir. Biyometri sadece kullanıcının cihaz üzerinde doğrulanmasını sağlar.

Yanlış kullanım şöyle olur:

```kotlin
if (biometricSuccess) {
    isLoggedIn = true
    openBalanceScreen()
}
```

Bu UI-only biometric olur. Yani biyometri sadece ekranı açan bir boolean kontrolü gibi kullanılmıştır.

Problem şu: rootlu cihazda saldırgan bu boolean değeri manipüle edebilir veya `openBalanceScreen()` fonksiyonunu direkt çağırabilir. Biyometrik doğrulamayı gerçekten geçmeden ekran açılabilir.

## Crypto-bound Biometric

Daha doğru kullanımda biyometri, Keystore içindeki bir anahtarın kullanımına bağlanır.

Yani mantık şu değildir:

```text
Biyometri başarılı → giriş yap
```

Daha doğru mantık şudur:

```text
Biyometri başarılı → Keystore key kullanılabilir → token decrypt edilir → server token'ı doğrular → giriş devam eder
```

Keystore key oluşturulurken `setUserAuthenticationRequired(true)` gibi bir ayar kullanılır. Bu ayar şunu söyler:

```text
Bu anahtar kullanıcı doğrulaması olmadan kullanılamaz.
```

Burada biyometri doğrudan login kararı vermez. Sadece token'ı çözen crypto key'in kullanılabilmesini sağlar.

Saldırgan sadece “biometric success” callback'ini taklit ederse bu tek başına anahtarı ele geçirmeye yetmez. Çünkü anahtar uygulamaya verilmez, Keystore içinde kalır. Ama token decrypt edildikten sonra RAM'de yakalanabilir. Yani yine rootlu cihaz senaryosunda sınırları bilmek gerekiyor.

Benim anladığım en net cümle şu:

```text
Yanlış: biometric okeyse giriş yap.
Doğru: biometric okeyse decrypt işlemini başlat, sonra server doğrulasın.
```

## Brute-force

Brute-force, saldırganın çok sayıda şifre denemesi yapmasıdır.

```text
user@example.com : 123456
user@example.com : password
user@example.com : qwerty
user@example.com : 111111
```

Bunu client tarafında çözmeye çalışmak yeterli değildir. Çünkü saldırgan Android uygulamasını kullanmak zorunda değil. Direkt backend'e curl, Postman veya script ile `/login` isteği atabilir.

Yanlış yaklaşım:

```text
Android app içinde 3 yanlış denemeden sonra login butonunu kapat.
```

Bu sadece UI kontrolüdür. Saldırgan app'i bypass edip API'ye gider.

Doğru yaklaşım server tarafındadır.

## Rate Limiting

Rate limiting, belirli bir süre içinde yapılabilecek istek sayısını sınırlamaktır.

Örnek:

```text
Aynı IP'den 1 dakikada en fazla 10 login denemesi.
Aynı kullanıcı hesabı için 5 hatalı denemeden sonra bekleme.
```

Sadece IP bazlı limit de tek başına yeterli olmayabilir. Çünkü saldırgan farklı IP'ler kullanabilir. Bu yüzden server genelde IP, kullanıcı hesabı, cihaz bilgisi, hata sayısı ve zaman aralığı gibi birden fazla sinyale bakmalıdır.

## Account Lockout

Account lockout, çok fazla hatalı login denemesinden sonra hesabı geçici olarak kilitlemektir.

Örnek:

```text
5 yanlış şifre → 5 dakika bekleme
10 yanlış şifre → ek doğrulama
Çok şüpheli durumda → kullanıcıya uyarı
```

Ama bunu çok agresif yapmak da risklidir. Saldırgan başkalarının e-posta adresleriyle sürekli yanlış deneme yapıp hesapları kilitleyebilir. Bu da DoS gibi kullanılabilir.

Bu yüzden kademeli bekleme, risk bazlı kontrol ve ek doğrulama daha mantıklı olabilir.

## User Enumeration

User enumeration, hata mesajlarından bir kullanıcının sistemde kayıtlı olup olmadığını anlamaktır.

Kötü örnek:

```text
Bu e-posta kayıtlı değil.
```

Başka kötü örnek:

```text
Şifre yanlış.
```

Bu iki mesaj saldırgana bilgi verir. Saldırgan hangisinin gerçek kullanıcı olduğunu anlayabilir.

Daha güvenli mesaj:

```text
Kullanıcı adı veya şifre hatalı.
```

Bu mesaj hem kullanıcı yokken hem de şifre yanlışken aynı dönmelidir.

## Logout

Logout sadece client tarafında login ekranına dönmek değildir.

Yanlış yaklaşım:

```text
Local token'ı sil.
Login ekranına dön.
```

Bu yeterli değildir. Çünkü saldırgan token'ı daha önce ele geçirmiş olabilir. Eğer server bu token'ı hâlâ kabul ediyorsa logout gerçek anlamda işe yaramaz.

Doğru logout iki taraflıdır:

```text
Server tarafında refresh token/session invalidate edilir.
Client tarafında local token ve hassas veriler silinir.
```

Özellikle refresh token logout sırasında geçersiz hale getirilmelidir. Çünkü refresh token çalışmaya devam ederse saldırgan yeni access token almaya devam edebilir.

Logout sonrası beklenen şey:

```text
Eski refresh token ile /refresh çalışmamalı.
Eski token ile korumalı endpoint'lere erişim olmamalı veya access token kısa sürede expire olmalı.
Client tarafında access token, refresh token, cached profile/balance gibi hassas veriler temizlenmeli.
```

## Rootlu Cihaz Notu

Bu Phase 1 konularında tekrar tekrar aynı noktaya geldim: rootlu cihaz kadar tehlikeli çok az senaryo var.

Normal cihazda Android sandbox, app private storage, Keystore ve biyometri ciddi koruma sağlar. Ama cihaz rootluysa saldırgan sadece dosya okumaz; uygulamanın runtime davranışına da müdahale edebilir.

Bu yüzden mobil güvenlikte savunmaları şöyle düşünmek daha doğru:

```text
Bu önlem saldırıyı imkansız yapıyor mu?
Hayır.

Bu önlem hangi saldırgan tipine karşı maliyeti artırıyor?
Bunu sormak daha doğru.
```

EncryptedSharedPreferences diskten token çalmayı zorlaştırır. Crypto-bound biometric UI-only bypass riskini azaltır. Rate limiting brute-force'u server tarafında sınırlar. Logout invalidation çalınmış refresh token'ın uzun süre kullanılmasını engeller. Ama tamamen saldırgan kontrolündeki rootlu cihaz için başka katmanlar da gerekir.

## Kısa Özet

Phase 1'de benim anladığım ana mantık şu:

```text
Login kararını server verir.
Client sadece server'ın verdiği token ile hareket eder.
Access token kısa ömürlü olur.
Refresh token daha kritik korunur.
JWT okunabilir ama imza olmadan güvenli şekilde değiştirilemez.
Token plain SharedPreferences içinde saklanmaz.
Keystore-backed encrypted storage kullanılır.
Biyometri sadece UI kapısı olmaz, crypto key kullanımına bağlanır.
Brute-force koruması server tarafında yapılır.
Hata mesajları user enumeration yaratmaz.
Logout hem server-side invalidation hem local cleanup demektir.
Rootlu cihazda hiçbir client-side önlem tek başına mutlak koruma değildir.
```
