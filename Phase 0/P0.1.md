# P0.1 — Review Keywords: HTTPS, Burp, JWT, Certificate Trust

## Amaç

Bu bölümde mentor feedback'inde geçen HTTPS haberleşmesi, Burp Suite ile request/response inceleme, JWT, Certificate Authority, self-signed certificate ve certificate pinning kavramları incelenmiştir.

Bu bölümün amacı tek tek bağımsız deney yapmak değil, Phase 1–2–3 sırasında küçük mock banking app üzerinde test edeceğimiz kavramları önceden netleştirmektir.

## Mentor Feedback Notları

- Client HTTPS haberleşmesi incelenecek: login ve veri görüntüleme istekleri
- Request body Burp Suite'te görünüyor mu?
- Response body Burp Suite'te görünüyor mu?
- Response değiştirilebilir mi?
- HTTPS'e rağmen trafik nasıl görünüyor?
- User-installed CA ve default Certificate Authority davranışı incelenecek
- JWT token request/response içinde nasıl taşınıyor?
- Self-signed SSL ile test ortamı kurulacak
- Certificate pinning ile Burp MITM engellenebilir mi?

## Temel Cevap

HTTPS trafiğinin Burp Suite'te görünüp görünmemesi client'ın sertifika güven kararına bağlıdır.

- Eğer uygulama Burp CA sertifikasına güveniyorsa, HTTPS olmasına rağmen request ve response içerikleri Burp'te görülebilir.
- Eğer uygulama Burp CA sertifikasına güvenmiyorsa, TLS bağlantısı başarısız olur veya içerik okunamaz.
- Eğer certificate pinning varsa, Burp CA yüklü olsa bile trafik genelde okunamaz; bağlantı pin verification hatasıyla başarısız olur.

## TLS ve HTTPS

TLS, HTTPS'in arkasındaki güvenlik protokolüdür. Amacı iki ana noktadır:

1. Trafiği şifrelemek
2. Client'ın gerçekten doğru server'a bağlandığını doğrulamak

Yani uygulama `https://api.example.com` adresine bağlanırken sadece veriyi şifrelemez; aynı zamanda karşısındaki server'ın gerçekten `api.example.com` olup olmadığını da kontrol eder.

Bu kontrol server sertifikası üzerinden yapılır.

## CA Mantığı

CA, **Certificate Authority** demektir. Bir server HTTPS kullanırken client'a bir sertifika gösterir. Bu sertifika temelde şunu kanıtlamaya çalışır:

```text
Ben gerçekten api.example.com'um.
```

Client bu söze direkt inanmaz. Şuna bakar:

```text
Bu sertifika güvenilir bir CA tarafından imzalanmış mı?
```

Android ve tarayıcılar bazı büyük CA'lara varsayılan olarak güvenir. Bunlara **system CA** denir.

Burp Suite de kendi CA sertifikasını üretir. Bu sertifika cihaza yüklendiğinde cihaz şunu kabul etmiş olur:

```text
Burp tarafından imzalanan sertifikalara da güvenebilirim.
```

## Burp CA Yüklüyse HTTPS İçeriği Neden Görünür?

Normalde bağlantı şöyle olur:

```text
Android App  ↔  Backend API
```

Burp proxy kullanınca bağlantı ikiye bölünür:

```text
Android App  ↔  Burp  ↔  Backend API
```

Burp burada ortadaki kişi olur. Uygulama backend'e bağlandığını sanır ama aslında önce Burp'e bağlanır. Burp uygulamaya hedef domain için sahte bir sertifika gösterir.

Eğer uygulama Burp CA'ya güveniyorsa bu sahte sertifikayı kabul eder. Böylece Burp, uygulama ile kendi arasında ayrı bir TLS bağlantısı, backend ile kendi arasında ayrı bir TLS bağlantısı kurar.

Sonuç:

```text
App ↔ Burp arasında HTTPS vardır.
Burp ↔ Server arasında HTTPS vardır.
Ama Burp ortada olduğu için içeriği plaintext görebilir.
```

Bu yüzden şu cümle doğrudur:

```text
HTTPS kullanılmasına rağmen, client Burp CA'ya güveniyorsa Burp request ve response body'lerini görebilir.
```

## Burp CA'ya Güvenilmiyorsa Ne Olur?

Client Burp tarafından sunulan sertifikayı güvenilir bulmazsa TLS bağlantısı kurulmaz.

Akış şu şekilde olur:

```text
App: api.example.com'a bağlanmak istiyorum.
Burp: Ben api.example.com'um, işte sertifikam.
App: Bu sertifika güvendiğim bir CA tarafından imzalanmamış.
App: Bağlantıyı reddediyorum.
```

Bu durumda login request body, Authorization header, JWT veya response body gibi içerikler Burp'te okunamaz.

## Request Body Burp'te Görünüyor mu?

Login isteği genelde kullanıcı adı ve şifre gibi hassas bilgiler içerir.

Örnek login request:

```http
POST /login HTTP/1.1
Host: api.mockbank.local
Content-Type: application/json

{
  "username": "intern",
  "password": "123456"
}
```

Burp Suite'te kontrol edilecek noktalar:

- `POST /login` isteği görünüyor mu?
- Request header'ları görünüyor mu?
- `Content-Type: application/json` mı?
- Request body tam görünüyor mu?
- Username ve password plaintext olarak okunabiliyor mu?

Beklenen sonuç:

- Burp CA trusted ise login request body plaintext görünebilir.
- Burp CA trusted değilse TLS bağlantısı başarısız olur ve body okunamaz.
- Certificate pinning varsa Burp CA trusted olsa bile body okunamaz.

## Response Body Burp'te Görünüyor mu?

Login başarılı olduğunda backend genelde client'a access token döner.

Örnek login response:

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "sample-refresh-token",
  "expiresIn": 900
}
```

Burp Suite'te kontrol edilecek noktalar:

- HTTP status code `200` mü?
- Response body görünüyor mu?
- `accessToken` alanı görünüyor mu?
- `refreshToken` alanı görünüyor mu?
- Token JWT formatında mı?
- Token süresi görünüyor mu?

Beklenen sonuç:

- Burp CA trusted ise login response body ve token bilgileri görülebilir.
- Burp CA trusted değilse response body okunamaz.
- Certificate pinning varsa login response Burp'te görüntülenemez.

## Response Burp ile Değiştirilebilir mi?

Burp Suite sadece request ve response'ları görüntülemek için kullanılmaz. Response içeriği client'a gönderilmeden önce değiştirilebilir.

Örneğin backend normalde şu response'u dönebilir:

```json
{
  "balance": 1250,
  "currency": "TRY",
  "isPremium": false
}
```

Burp üzerinden response şu şekilde değiştirilebilir:

```json
{
  "balance": 999999,
  "currency": "TRY",
  "isPremium": true
}
```

Bu durumda kontrol edilmesi gereken şey şudur:

```text
Client ekranda hangi veriyi gösteriyor?
Gerçek backend verisi mi gösteriliyor?
Yoksa Burp ile değiştirilen sahte response mu gösteriliyor?
```

Eğer client response'u doğrudan UI'a yansıtıyorsa kullanıcı uygulamada Burp ile değiştirilen değeri görebilir. Bu server'daki gerçek verinin değiştiği anlamına gelmez. Örneğin backend'deki gerçek bakiye hâlâ `1250` olabilir; sadece client'a gönderilen response değiştirilmiştir.

Bu test özellikle client'ın güvenlik kararlarını response body'deki alanlara göre verip vermediğini anlamak için önemlidir. Örneğin `role`, `isAdmin`, `isPremium`, `limit` gibi alanlara client tarafında körü körüne güvenilmesi risklidir.

## Burp'te Response Nasıl Değiştirilir?

En basit yöntem Intercept ile response'u yakalayıp elle değiştirmektir.

1. Burp'te `Proxy → Intercept → Intercept is on` yapılır.
2. Uygulamada `/balance` gibi response döndüren bir işlem tetiklenir.
3. Request Burp'e düşünce request üzerinde sağ tıklanır.
4. `Do intercept → Response to this request` seçilir.
5. Request `Forward` edilir.
6. Backend'den dönen response Burp'te yakalanır.
7. Response body içindeki değer değiştirilir.
8. Değiştirilmiş response tekrar `Forward` edilir.

Örnek:

```text
Backend gerçek balance: 1250
Burp ile client'a gönderilen balance: 999999
Uygulamada görünen balance: 999999
```

Alternatif olarak `Proxy → Match and Replace` ile belirli response alanları otomatik değiştirilebilir.

## User-installed CA ve Default Certificate Authority Davranışı

Android uygulamaları HTTPS bağlantısı kurarken server'ın sunduğu sertifikanın güvenilir olup olmadığını kontrol eder. Bu kontrol sırasında uygulamanın hangi Certificate Authority'lere güvendiği önemlidir.

Genel olarak iki farklı CA kaynağı vardır:

```text
1. System CA
2. User-installed CA
```

**System CA**, Android işletim sistemiyle birlikte gelen ve varsayılan olarak güvenilen sertifika otoriteleridir. Normal bir HTTPS bağlantısında uygulama, server sertifikasının bu güvenilir CA'lardan biri tarafından imzalanıp imzalanmadığını kontrol eder.

**User-installed CA**, kullanıcının veya test yapan kişinin sonradan cihaza yüklediği CA sertifikasıdır. Burp Suite'in CA sertifikası buna örnektir.

Burp CA cihaza yüklendiğinde cihaz şunu kabul etmiş olur:

```text
Burp tarafından imzalanan sertifikalara da güvenebilirim.
```

Bu durum test ortamında HTTPS trafiğini incelemek için kullanılır. Burp hedef domain için sahte bir sertifika üretir. Eğer client Burp CA'ya güveniyorsa bu sahte sertifika kabul edilir ve Burp HTTPS trafiğini plaintext olarak gösterebilir.

Production ortamında uygulamanın user-installed CA'lara güvenmemesi tercih edilir. Böylece kullanıcı cihaza Burp CA gibi bir sertifika yüklese bile uygulama sadece system CA'lara güvenir ve Burp'ün ürettiği sahte sertifikayı kabul etmez.

Örnek production yaklaşımı:

```xml
<base-config cleartextTrafficPermitted="false">
    <trust-anchors>
        <certificates src="system" />
    </trust-anchors>
</base-config>
```

Bu yapı şunu söyler:

```text
HTTP cleartext trafik yasak.
Sadece system CA'lara güven.
User-installed CA'lara güvenme.
```

Debug ortamında test yapabilmek için user CA'lara geçici olarak güven verilebilir:

```xml
<debug-overrides>
    <trust-anchors>
        <certificates src="user" />
    </trust-anchors>
</debug-overrides>
```

Bu durumda debug build Burp CA'ya güvenebilir ve HTTPS trafiği Burp Suite'te incelenebilir. Ancak bu ayar production build'e taşınmamalıdır.

## JWT Token Request/Response İçinde Nasıl Taşınıyor?

Mobil uygulamalarda yaygın akış şu şekildedir:

```text
1. Client username/password ile login isteği atar.
2. Backend bilgileri doğrular.
3. Backend response içinde access token döner.
4. Client bu token'ı saklar.
5. Client sonraki korumalı API isteklerinde token'ı Authorization header içinde gönderir.
```

Örnek login response:

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "sample-refresh-token",
  "expiresIn": 900
}
```

JWT üç parçadan oluşur:

```text
header.payload.signature
```

JWT'nin ilk iki parçası çoğu zaman Base64URL ile encode edilmiştir. Bu yüzden token ele geçirilirse içeriği decode edilerek okunabilir.

Örnek payload:

```json
{
  "sub": "123",
  "role": "user",
  "exp": 1719999999
}
```

Önemli nokta:

```text
JWT çoğu zaman şifreli değildir, imzalıdır.
```

Yani token'ın içeriği okunabilir ama imza için kullanılan secret/private key bilinmeden geçerli şekilde değiştirilemez.

Login sonrası client bu token'ı korumalı API isteklerinde genellikle Authorization header içinde taşır.

Örnek balance request:

```http
GET /balance HTTP/1.1
Host: api.mockbank.local
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Accept: application/json
```

Burp Suite'te kontrol edilecek noktalar:

- Login response içinde `accessToken` görünüyor mu?
- Token JWT formatında mı?
- Refresh token da response içinde geliyor mu?
- Sonraki isteklerde `Authorization` header var mı?
- `Authorization: Bearer ...` header'ı görünüyor mu?
- Token decode edildiğinde hassas veri var mı?
- Token süresi çok uzun mu?

Güvenlik açısından burada iki farklı risk vardır:

1. Token'ın ağ üzerinde ele geçirilmesi
2. Token'ın cihaz üzerinde güvensiz saklanması

Eğer token ele geçirilirse saldırgan kullanıcının şifresini bilmeden onun adına API istekleri gönderebilir. Bu yüzden access token kısa ömürlü olmalı, refresh token daha dikkatli korunmalı ve logout durumunda token server tarafında da geçersizleştirilmelidir.

## Self-Signed SSL ile Test Ortamı Kurarsak Ne Olur?

Self-signed sertifikada sertifikayı dışarıdan güvenilir bir CA imzalamaz. Sertifika kendi kendini imzalar.

Normal sertifika zinciri şöyleyken:

```text
Trusted Root CA
    ↓
Intermediate CA
    ↓
api.example.com sertifikası
```

Self-signed sertifikada yapı daha çok şöyle olur:

```text
api.local sertifikası
    ↓
Kendi kendini imzalar
```

Bu yüzden Android client default olarak bu sertifikaya güvenmez.

Beklenen davranış:

```text
Self-signed HTTPS backend'e istek atılırsa:
Client sertifikaya güvenmez.
TLS handshake başarısız olur.
Connection error veya certificate error alınır.
Request/response body okunamaz.
```

Bu davranış güvenlik açısından doğrudur. Çünkü herkes kendi kendine sertifika üretebilir. Client her self-signed sertifikaya güvenseydi, saldırgan da sahte sertifika üretip kendisini backend gibi gösterebilirdi.

Test ortamında self-signed sertifikayı kullanmak istiyorsak client'a bu sertifikaya bilinçli olarak güvenmesini söylememiz gerekir. Android tarafında bu genellikle Network Security Config ile yapılabilir.

Debug ortamı için özel bir CA sertifikası eklenebilir:

```xml
<network-security-config>
    <debug-overrides>
        <trust-anchors>
            <certificates src="@raw/debug_ca" />
        </trust-anchors>
    </debug-overrides>
</network-security-config>
```

Bu yapı şunu söyler:

```text
Debug build sırasında @raw/debug_ca sertifikasına güven.
```

Production build içinde rastgele self-signed CA veya debug CA bırakılmamalıdır. Aksi halde uygulama gereksiz yere sahte sertifikalara güvenebilir hale gelir.

## Certificate Pinning Nedir?

Certificate pinning, uygulamanın sadece CA güvenine bakmaması ve beklenen sertifika veya public key değerini ayrıca doğrulamasıdır.

Normal HTTPS kontrolü şunu sorar:

```text
Bu sertifika güvenilir bir CA tarafından imzalanmış mı?
Domain doğru mu?
Sertifika süresi geçerli mi?
```

Pinning ek olarak şunu sorar:

```text
Gelen sertifika veya public key benim beklediğim pin değeriyle eşleşiyor mu?
```

Örnek:

```text
Beklenen server pin: sha256/ABC123
Burp sertifikasının pin'i: sha256/XYZ999
Sonuç: Pin eşleşmediği için bağlantı reddedilir.
```

Bu yüzden Burp CA cihaza yüklü olsa bile, certificate pinning aktifse Burp trafiği açamaz.

## Pinning Türleri

### Leaf Certificate Pinning

Direkt server'ın kendi sertifikasını pinlemektir.

Avantajı: Dar ve güçlü kontroldür.

Dezavantajı: Server sertifikası yenilenince pin bozulabilir ve uygulama bağlantı kuramaz hale gelebilir.

### SPKI / Public Key Pinning

Tüm sertifika yerine sertifikanın içindeki public key pinlenir.

Avantajı: Sertifika yenilense bile aynı key pair kullanılırsa pin bozulmaz. Burp gibi sahte sertifikaları engeller.

Dezavantajı: Private key değişirse pin de değişir. Bu yüzden backup pin gerekir.

### Intermediate CA Pinning

Server'ın kendi sertifikası değil, onu imzalayan ara CA pinlenir.

Avantajı: Leaf sertifika yenilense bile aynı intermediate CA kullanılıyorsa bağlantı bozulmaz.

Dezavantajı: Leaf/SPKI pinning kadar dar değildir. Aynı intermediate CA başka sertifikalar da verebilir.

### Root CA Pinning

Sadece belirli bir root CA zincirine güvenmektir.

Avantajı: Sertifika yenilemede daha az sorun çıkarır.

Dezavantajı: Çok geniştir. Güvenlik katkısı daha düşüktür çünkü o root CA altında çok fazla sertifika üretilebilir.

## Pinning Nasıl Bypass Edilir?

Burada iki saldırgan türünü ayırmak gerekir.

### Ağ Saldırganı

Bu saldırgan sadece Wi-Fi/router/proxy seviyesinde araya girer.

Elinde şunlar yoktur:

- Root erişimi
- Uygulama memory'sine erişim
- Frida
- Gerçek server private key'i

Bu saldırgan için pinning çok etkilidir. Burp CA yüklenmiş olsa bile pin uyuşmadığı için bağlantı başarısız olur.

### Rootlu Cihaz Saldırganı

Bu saldırgan cihazı kontrol eder.

Elinde şunlar olabilir:

- Root erişimi
- Frida
- Objection
- Runtime hook
- APK patchleme
- Repackaging

Bu durumda saldırgan pinning'i kriptografik olarak kırmak zorunda değildir. Bunun yerine uygulamanın pin kontrolünü runtime'da devre dışı bırakabilir.

Mantık:

```text
Normal:
checkCertificatePin() → false
Bağlantı reddedilir.

Hook sonrası:
checkCertificatePin() → true
Bağlantı kabul edilir.
```

Objection örneği:

```bash
objection -g com.paket.adi explore
android sslpinning disable
```

Bu yüzden pinning ağ seviyesindeki MITM saldırganına karşı güçlü bir savunmadır, ancak rootlu veya tamamen kontrol edilen cihazda tek başına mutlak koruma sağlamaz.

## Sık Sorulan Sorular

### Büyük güvenilir CA'yı kopyalayıp kendi app'ime koyamaz mıyım?

Koyabilirsin ama bu sana saldırı gücü kazandırmaz.

Public CA certificate herkes tarafından görülebilir. Ancak CA private key sadece CA sahibindedir. Bir sertifikanın gerçekten o CA tarafından imzalanmış sayılması için CA'nın private key'i gerekir. Sadece public sertifikayı kopyalayarak yeni geçerli sertifika üretemezsin.

### Gerçek sitenin pinini alıp Burp'te aynı pini kullansam olmaz mı?

Normalde olmaz. Çünkü pin genelde sertifikanın veya public key'in hash'idir. Sen gerçek server'ın pinini öğrenebilirsin ama Burp araya girdiğinde app'e Burp'ün oluşturduğu sertifika gösterilir. Bu sertifikanın public key'i farklıdır.

Burp'ün gerçek server ile aynı pini kullanabilmesi için gerçek server'ın private key'ine sahip olması gerekir. Bu normal şartlarda mümkün değildir.

### Sertifika içeriğini kopyalasam olmaz mı?

Olmaz. TLS sırasında server sadece sertifika metnini göstermez; aynı zamanda bu sertifikanın public key'ine karşılık gelen private key'e sahip olduğunu kanıtlar. Sertifikayı kopyalamak private key'e sahip olmak anlamına gelmez.

## Kısa Sonuç

Bu review kısmında HTTPS, Burp, CA trust, JWT, self-signed certificate ve certificate pinning kavramları netleştirildi. Bu kavramlar ayrı ayrı bağımsız deneyler olarak değil, küçük mock banking app üzerinde login ve balance/profile akışı üzerinden test edilecektir.

Temel test fikri:

```text
1. Burp CA trusted + pinning yok → request/response/JWT görünür, response değiştirilebilir.
2. Burp CA trusted değil → TLS başarısız olur, içerik görünmez.
3. Pinning aktif → Burp CA trusted olsa bile bağlantı reddedilir.
4. Root + Frida/objection → pinning bypass denenebilir.
```
